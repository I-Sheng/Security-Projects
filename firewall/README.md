# Firewall

This module covers Linux firewall implementation from the ground up: writing kernel modules, hooking into the Netfilter framework, configuring stateless and stateful `iptables` rules, rate limiting, and DNAT-based load balancing.

## Prerequisites

- Linux host with kernel headers installed (`apt install linux-headers-$(uname -r)`)
- Root / `sudo` privileges
- `iptables`, `conntrack`, `macchanger` available
- Docker (optional, for the multi-host lab topology)

## Lab Tasks

### Task 1 — Implementing a Firewall with Netfilter

#### Task 1A — Hello World Kernel Module

A minimal kernel module that prints to the kernel log on load and unload.

```c
#include <linux/module.h>
#include <linux/kernel.h>

int initialization(void) { printk(KERN_INFO "Hello World!\n"); return 0; }
void cleanup(void)        { printk(KERN_INFO "Bye-bye World!.\n"); }

module_init(initialization);
module_exit(cleanup);
```

**Makefile:**

```makefile
obj-m += hello.o
all:
    make -C /lib/modules/$(shell uname -r)/build M=$(PWD) modules
clean:
    make -C /lib/modules/$(shell uname -r)/build M=$(PWD) clean
```

**Load and verify:**

```bash
sudo insmod hello.ko
lsmod | grep hello
sudo rmmod hello
dmesg          # shows "Hello World!" and "Bye-bye World!"
```

#### Task 1B — Netfilter Hooks

A kernel module registers two Netfilter hooks:

- **`NF_INET_LOCAL_OUT` hook (`printInfo`):** Logs every locally-generated IPv4 packet, showing the active hook point, source/destination IPs, and protocol (UDP/TCP/ICMP/other).
- **`NF_INET_POST_ROUTING` hook (`blockUDP`):** Drops UDP packets destined for `8.8.8.8:53`, blocking DNS queries to Google's resolver.

**Netfilter hook points (in packet flow order):**

| Hook | Position in packet path |
|---|---|
| `NF_INET_PRE_ROUTING` | After entering the IP stack, before routing decision |
| `NF_INET_LOCAL_IN` | Routing decided this host is the destination |
| `NF_INET_FORWARD` | Routing decided the packet should be forwarded |
| `NF_INET_LOCAL_OUT` | Packet generated by a local process, before routing |
| `NF_INET_POST_ROUTING` | After routing/NAT, just before leaving the interface |

**Blocking Telnet (TCP port 23) and ICMP:**

```c
// Block TCP to 10.9.0.1:23
unsigned int blockTelnet(void *priv, struct sk_buff *skb,
                         const struct nf_hook_state *state) {
    struct iphdr *iph = ip_hdr(skb);
    struct tcphdr *tcph;
    u32 ip_addr;
    in4_pton("10.9.0.1", -1, (u8 *)&ip_addr, '\0', NULL);
    if (iph->protocol == IPPROTO_TCP) {
        tcph = tcp_hdr(skb);
        if (iph->daddr == ip_addr && ntohs(tcph->dest) == 23)
            return NF_DROP;
    }
    return NF_ACCEPT;
}

// Block ICMP to 10.9.0.1
unsigned int blockICMP(void *priv, struct sk_buff *skb,
                       const struct nf_hook_state *state) {
    struct iphdr *iph = ip_hdr(skb);
    u32 ip_addr;
    in4_pton("10.9.0.1", -1, (u8 *)&ip_addr, '\0', NULL);
    if (iph->protocol == IPPROTO_ICMP && iph->daddr == ip_addr)
        return NF_DROP;
    return NF_ACCEPT;
}
```

### Task 2 — Stateless `iptables` Rules

#### Task 2A — Protecting the Router

Allow only ICMP (ping) in and out; drop everything else:

```bash
iptables -A INPUT  -p icmp --icmp-type echo-request -j ACCEPT
iptables -A OUTPUT -p icmp --icmp-type echo-reply   -j ACCEPT
iptables -P OUTPUT DROP
iptables -P INPUT  DROP
```

Result: the router responds to pings but rejects all TCP connections.

---

#### Task 2B — Protecting the Internal Network

Topology: external hosts on `10.9.0.0/24`, internal hosts on `192.168.60.0/24`, router bridging both via `eth0` (external) and `eth1` (internal).

Policy: internal hosts can ping externally; external hosts cannot ping internal hosts directly.

```bash
iptables -P FORWARD DROP
iptables -A FORWARD -i eth1 -o eth0 -s 192.168.60.0/24 -p icmp --icmp-type echo-request -j ACCEPT
iptables -A FORWARD -i eth0 -o eth1 -d 192.168.60.0/24 -p icmp --icmp-type echo-reply   -j ACCEPT
iptables -A FORWARD -i eth0 -o eth1 -d 192.168.60.0/24 -p icmp --icmp-type echo-request -j DROP
iptables -A INPUT   -i eth0 -p icmp --icmp-type echo-request -j ACCEPT
iptables -A OUTPUT  -o eth0 -p icmp --icmp-type echo-reply   -j ACCEPT
iptables -P INPUT  DROP
iptables -P OUTPUT DROP
```

#### Task 2C — Protecting Internal Servers

Only external hosts may Telnet to `192.168.60.5`; internal hosts may Telnet to each other freely.

```bash
iptables -P FORWARD DROP
# External → 192.168.60.5:23
iptables -A FORWARD -i eth0 -o eth1 -s 10.9.0.0/24 -d 192.168.60.5 -p tcp --dport 23 -j ACCEPT
# Return traffic
iptables -A FORWARD -i eth1 -o eth0 -s 192.168.60.5 -d 10.9.0.0/24 -p tcp --sport 23 -j ACCEPT
# Internal ↔ internal
iptables -A FORWARD -i eth1 -o eth1 -p tcp -j ACCEPT
```

### Task 3 — Stateful Firewall with `conntrack`

#### Task 3A — Connection Tracking Observations

`conntrack -L` tracks flows by protocol:

- **ICMP:** Request + reply grouped as one entry; both directions shown.
- **UDP:** Once traffic flows in both directions, the entry is marked `[ASSURED]` and persists ~2 minutes after the last packet.
- **TCP:** Full handshake tracked; state transitions from `SYN_SENT` → `ESTABLISHED`; entry is `[ASSURED]` after bidirectional data and lingers ~2 minutes after teardown.

**Monitoring conntrack in real time:**
```bash
watch -n 1 conntrack -L
```

---

#### Task 3B — Stateful Firewall Rules

```bash
iptables -P FORWARD DROP

# Allow new connections: external → 192.168.60.5:23 only
iptables -A FORWARD -i eth0 -o eth1 \
  -s 10.9.0.0/24 -d 192.168.60.5 -p tcp --dport 23 --syn \
  -m conntrack --ctstate NEW -j ACCEPT

# Allow new connections: internal → any external host
iptables -A FORWARD -i eth1 -o eth0 \
  -s 192.168.60.0/24 -d 10.9.0.0/24 -p tcp --syn \
  -m conntrack --ctstate NEW -j ACCEPT

# Allow packets belonging to established/related flows
iptables -A FORWARD -p tcp -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

**Stateless vs. Stateful comparison:**

| Aspect | Stateless (Task 2C) | Stateful (Task 3B) |
|---|---|---|
| Rule matching | Per-packet header fields only | Tracks connection state across packets |
| Return traffic | Requires explicit bidirectional rules | Covered automatically by `ESTABLISHED,RELATED` |
| Rule count | Grows with each service | Fewer, higher-level rules |
| Overhead | No conntrack table | Small conntrack overhead, negligible for typical loads |

### Task 4 — Rate Limiting

Limit forwarded traffic from `10.9.0.5` to 10 packets per minute, with a burst of 5:

```bash
iptables -A FORWARD -s 10.9.0.5 -m limit --limit 10/minute --limit-burst 5 -j ACCEPT
iptables -A FORWARD -s 10.9.0.5 -j DROP
```

The second rule is essential: without it, packets that don't match the rate-limiter fall through to the default `ACCEPT` policy and are not dropped.

### Task 5 — Load Balancing with DNAT

Distribute incoming UDP traffic on port 8080 across three backend servers: `192.168.60.5`, `.6`, and `.7`.

**Round-robin (deterministic):**

```bash
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -m statistic --mode nth --every 3 --packet 0 -j DNAT --to-destination 192.168.60.5:8080
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -m statistic --mode nth --every 3 --packet 1 -j DNAT --to-destination 192.168.60.6:8080
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -m statistic --mode nth --every 3 --packet 2 -j DNAT --to-destination 192.168.60.7:8080
```

**Random (probabilistic):**

```bash
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -m statistic --mode random --probability 0.33 -j DNAT --to-destination 192.168.60.5:8080
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -m statistic --mode random --probability 0.5  -j DNAT --to-destination 192.168.60.6:8080
iptables -t nat -A PREROUTING -p udp --dport 8080 \
    -j DNAT --to-destination 192.168.60.7:8080
```

With only 50 test packets the observed distribution deviates from a perfect 1/3 split due to small sample size and UDP's unreliability.
